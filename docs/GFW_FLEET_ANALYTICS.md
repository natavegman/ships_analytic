# GFW Fleet Analytics — эффективность флота и продажи оборудования

Модуль аналитики на основе выгрузок Global Fishing Watch (CSV events + tracks) с интеграцией официального вылова (АДМ) и классовых требований РС (RMRS).

**Статус:** реализовано на примере флота НБАМР (7 судов, 6 БМРТ + 1 плавзавод).  
**Дата среза:** июнь 2026.

---

## Зачем

GFW API (без Advanced) недостаточен для сырых треков — используем **ручные CSV-выгрузки событий**. Цель: точная картина эксплуатации судна (тралёжка, переход, простой, перегруз, порт, ремонт) + **фактический вылов** + прогноз обслуживания для продаж оборудования.

**Ключевой инсайт:** GFW-активность ≠ вылов. Пример: КАПИТАН ФАЛЕЕВ — №1 по вылову НБАМР (8492 т минтая), но низкая GFW-интенсивность (24.5%). Ранжировать флот только по GFW нельзя.

## Скрипты

| Скрипт | Назначение |
|--------|------------|
| `scripts/gwf_analytic.py` | Бюджет активности по сырым трекам (траление / переход / простой), детекция валов |
| `scripts/gfw_repairs_analytic.py` | Ремонты и докования из `port_visit`, геотег верфей, цикл докования, RMRS |
| `scripts/gfw_fleet_analytic.py` | **Главный:** флотовые KPI, encounter-классификация, бенчмарк, scorecard |
| `scripts/gfw_catch_cargo.py` | Вылов × GFW: продуктивность, набор груза → перегруз |
| `scripts/gfw_class_survey.py` | Классовые освидетельствования РС, forward-прогноз докования |
| `scripts/fetch_gfw_tracks_batch.py` | Пакетная загрузка треков (fallback на hourly AIS) |
| `scripts/gfw_client.py` | Клиент GFW API (треки, presence) |

### Запуск (НБАМР)

```bash
cd Quotas_analytic
.venv/bin/python scripts/gfw_fleet_analytic.py --input-dir data/nbamr_events --rmrs-dir output
```

### Выход (`output/gfw_fleet/`)

- `fleet_scorecard.csv` — KPI по судну + вылов + набор груза + прогноз обслуживания
- `fleet_benchmark.csv` — ранги **внутри peer_group** (БМРТ отдельно от плавзавода)
- `fleet_yearly.csv` — бюджет времени судно×год
- `encounters.csv` — встречи в море с классификацией

## Методология

### Типы судов и сегментация

- **БМРТ** (Беляков, Фалеев, Коновалов, Сивинд-1, Чепик, Масловец) — морозят Б/Г+икру, сдача в море ~80%, каденция перегрузов **12–17 сут**, набор за рейс **~480–670 т**.
- **ЦАРИЦА** — **плавзавод (СТР/РТМКС)**, выпускает минтай НР. **Не сравнивать с БМРТ** (другая логистика: 26% сдачи в море, каденция 6 сут). `peer_group=factory`, вне рейтинга БМРТ.

### KPI эффективности (GFW)

- `fishing_intensity_%` — промысел / наблюдаемые дни
- `deployment_eff_%` — промысел / дни в море
- `availability_%` — (год − ремонт − отстой) / год
- `autonomy_days` — медиана интервала между заходами в порт
- `at_sea_offload_share_%` — перегрузы / (перегрузы + выгрузки в порту)

### Encounter-классификация

- `transshipment` — рефрижератор по имени или контакт ≥6 ч
- `bunkering` — танкер по ключевым словам
- `supply_other` — прочее

### Набор груза → перегруз (`gfw_catch_cargo.py`)

- `catch_per_seaday_t` = вылов_сезона / at_sea_days (GFW 2026)
- `offload_cadence_days` = медиана внутрисезонных интервалов между перегрузами (вся история GFW, паузы >90 сут отбрасываются)
- `effective_hold_t` = паспортный трюм ИЛИ `catch_per_seaday_t × cadence`
- `days_to_fill_hold` = effective_hold / catch_per_seaday_t

### Ремонты (`gfw_repairs_analytic.py`)

Таксономия `port_visit`: `dry_dock_repair`, `repair_docking`, `extended_layup`, `unloading_call`, `bunker_call`. Геотег против TrustedDocks + curated yards (30 км).

### Классовые осмотры РС — **без «overdue»**

**Правило:** судно в эксплуатации ⇒ класс РС действителен ⇒ просроченного обязательного докования быть не может. Если расчётная дата в прошлом — окно прокатывается вперёд (операционный цикл и классовое докование). Ежегодные осмотры «на плаву» **не** формируют окно продаж.

Статусы: `due_soon` (≤3 мес), `approaching` (≤12 мес), `scheduled`.

### efficiency_score (0–100)

Нормировка **внутри peer_group**. Веса для БМРТ: вылов/сутки в море (0.30), интенсивность (0.25), готовность (0.20), развёртывание (0.15), автономность (0.10).
