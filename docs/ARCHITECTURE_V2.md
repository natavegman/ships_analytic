ТЗ: Миграция системы Quotas_analytic на DWH (PostgreSQL)
📌 1. Контекст и Цели (Для ИИ-ассистента)
Текущее состояние: Проект состоит из набора независимых Python-скриптов (ETL), которые парсят квоты (etl_quota.py), реестр Цербер (cerberus_download_auto.py), ФНС (enrich_company_groups...) и сохраняют данные в CSV-файлы и напрямую в базы Notion.
Проблема: При росте объема данных (добавление ежедневных выловов АМП, трекинга GFW и финансовой аналитики ВЭД) Notion не справляется с ролью базы данных. Невозможно реализовать сложную логику перекрестных связей (например, облов квот через тайм-чартер чужими судами).
Целевое состояние: 1. Внедрить PostgreSQL как центральное хранилище данных (DWH).
2. Написать логику "Трансферов квот", которая динамически подменяет владельца улова в зависимости от дат аренды судна.
3. Добавить финансовый модуль (UN Comtrade API).
4. Перевести Notion исключительно в режим Read-Only дашборда для конечных пользователей.

🛠 2. Стек технологий
Язык: Python 3.10+

БД: PostgreSQL (взаимодействие через SQLAlchemy 2.0+ и Alembic для миграций).

ETL/Оркестрация: n8n (вызов скриптов по cron).

API: Notion API, Global Fishing Watch API, UN Comtrade API.

🏗 3. Архитектура Базы Данных (Схема для SQLAlchemy)
ИИ-ассистенту необходимо создать следующие базовые таблицы:

companies (ИНН, Название, Группа_Компаний, Роль_в_холдинге)

vessels (IMO, Название, Проект, Базовый_Владелец_ИНН, GFW_ID)

quotas_limits (Год, ИНН_Владельца, Бассейн, Объект_Лова, Объем_Тонн)

quota_transfers (Ключевая таблица аренды)

id (PK)

vessel_id (FK -> vessels)

actual_quota_owner_inn (FK -> companies)

start_date (Date)

end_date (Date)

daily_catches (Фактический вылов)

date (Date)

vessel_id (FK -> vessels)

volume_tons (Float)

source (Enum: 'AMP', 'GFW_CALC', 'MANUAL')

market_prices (Цены ВЭД)

year_month (String/Date)

species (String)

price_usd_kg (Float)

🚀 4. План реализации (Спринты для Cursor)
Спринт 1: Фундамент БД и ORM
Задача: Настроить PostgreSQL и перевести текущие скрипты на запись в БД вместо CSV.

Шаг 1.1: Создать файл database/models.py с использованием SQLAlchemy по схеме из п.3.

Шаг 1.2: Настроить Alembic для управления миграциями (alembic init alembic).

Шаг 1.3: Отрефакторить скрипты etl_quota.py и enrich_company_groups... так, чтобы финальный датафрейм (pandas) записывался в базу Postgres через df.to_sql или сессии ORM.

Спринт 2: Модуль "Трансферы квот" (Бизнес-логика)
Задача: Научить систему понимать, кто реально владеет рыбой в момент вылова.

Шаг 2.1: Написать функцию-агрегатор calculate_vessel_catch(vessel_id, catch_date, volume).

Шаг 2.2: Логика: Функция делает SELECT в quota_transfers. Если на catch_date судно находилось в аренде у ИНН "X" — улов плюсуется к балансу ИНН "X". Если активного трансфера нет — улов плюсуется базовому владельцу судна.

Спринт 3: Новые коннекторы (АМП и Comtrade)
Задача: Добавить сбор сырца и финансовых показателей.

Шаг 3.1: Создать скрипт scripts/parser_amp.py для сбора графиков подходов в порты (BeautifulSoup/Playwright) и записи в daily_catches.

Шаг 3.2: Создать скрипт scripts/fetch_comtrade.py. Логика: Запрос к UN Comtrade Data API по кодам HS (030367 - минтай, 030614 - краб), импорт в Китай (партнер 156), экспорт из РФ (репортер 643). Запись средней цены в market_prices.

Спринт 4: Финансовый P&L и выгрузка в Notion
Задача: Свести тонны и деньги, обновить дашборды.

Шаг 4.1: Создать SQL View (представление) или Python-функцию generate_financial_report(), которая умножает консолидированный улов компании на среднюю цену из market_prices.

Шаг 4.2: Отрефакторить prepare_notion_import.py. Теперь он должен делать SELECT из PostgreSQL (только агрегированные данные: остаток квоты, расчетная выручка) и обновлять карточки в Notion через Notion API (метод PATCH).
